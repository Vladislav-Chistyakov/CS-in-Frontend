## AI конспект для лекции #12

## Введение: анатомия программы

Программа в своём простейшем виде — это последовательность инструкций и данных, сохранённая на диске. Но чтобы программа
заработала, операционная система должна загрузить её в память, выделить ресурсы и передать управление. Современная
программа — это не только код, но и стек, куча, глобальные переменные, а также контекст выполнения, который поддерживает
ОС.

## Многозадачность и изоляция процессов

На компьютере одновременно работают десятки программ: браузер, редактор кода, музыкальный плеер, мессенджер. Как они не
мешают друг другу? Как одна программа не может прочитать память другой (если только это не разрешено явно)?

Это одна из ключевых задач операционной системы — **изоляция** ресурсов. Каждая программа работает в своей песочнице и
не должна влиять на другие ни через память, ни через процессорное время.

## Процессы

**Процесс** — это экземпляр выполняющейся программы. У процесса есть:

- своё **адресное пространство** (набор виртуальных адресов, которые он может использовать, организованный в виде
  нескольких сегментов);
- один или несколько **потоков** (нитей выполнения);
- набор прав и идентификаторов (PID, пользователь, группа);
- контекст выполнения (значения регистров, состояние стека).

С точки зрения ОС, процесс — это единица владения ресурсами, но не обязательно единица выполнения.

## Межпроцессное взаимодействие (IPC)

Поскольку процессы изолированы, им нужны механизмы для обмена данными. Основные способы межпроцессного взаимодействия:

- **Каналы (pipes)** — однонаправленная передача данных между родственными процессами.
- **Именованные каналы (FIFO)** — работа через файловую систему.
- **Сигналы (signals)** — асинхронные уведомления (например, `SIGKILL`).
- **Сокеты (sockets)** — обмен данными между процессами на одной или разных машинах.
- **Разделяемая память (shared memory)** — самый быстрый способ, несколько процессов отображают одну область физической
  памяти в свои адресные пространства.
- **Очереди сообщений (message queues)**.

## Потоки (Threads)

Процесс может содержать несколько **потоков** — лёгких единиц выполнения, которые разделяют адресное пространство
процесса. У каждого потока есть свой стек и набор регистров, но общие код, глобальные данные, куча и файловые
дескрипторы.

**Преимущества потоков:**

- создание и переключение между потоками дешевле, чем между процессами;
- простота обмена данными — потоки одного процесса видят память друг друга.

**Недостатки:**

- один поток может разрушить данные всего процесса;
- нужна синхронизация (мьютексы, семафоры, критические секции).

## Адресное пространство процесса и стартовые регистры

Когда процесс создаётся, ОС не выделяет ему один непрерывный блок памяти. Вместо этого ОС создаёт для процесса **таблицы
страниц** и формирует **набор виртуальных областей** (VMA — Virtual Memory Areas в Linux, секции в Windows). Каждая
область имеет свои права доступа и назначение.

ОС создаёт отображения (memory mappings) для каждого сегмента из исполняемого файла (ELF/PE/Mach-O) плюс дополнительные
области для стека, кучи и разделяемых библиотек. Сами виртуальные адреса не обязаны быть непрерывными — процессор и MMU
скрывают разрывы через таблицы страниц, но с точки зрения программы адресное пространство *выглядит* непрерывным в
пределах доступных областей.

**Стартовые регистры** при запуске процесса (или нового потока) инициализируются следующим образом:

- **Указатель инструкции (RIP/EIP/IP)** устанавливается на точку входа. Однако это не обязательно `main`. На практике
  первым выполняется код **стартовой процедуры** (crt0, `_start`), которая настраивает среду (аргументы командной
  строки, окружение, инициализацию статических объектов в C++) и уже затем вызывает `main`.
- **Указатель стека (RSP/ESP/SP)** устанавливается на вершину только что выделенного стека. Стек — это отдельная
  виртуальная область со своими правами (обычно `rw-`), которая может динамически расти вниз. В конце стековой области
  часто размещается **защитная страница** (guard page) — при обращении к ней возникает исключение, что позволяет
  обнаружить переполнение стека.

## Виртуальная память

**Виртуальная память** — это абстракция, которая даёт каждому процессу иллюзию, что он один работает со всей памятью,
начиная с адреса `0`. Физическая память (ОЗУ) скрыта за этой абстракцией.

Как это работает:

- Процесс оперирует **виртуальными адресами**.
- ОС совместно с процессором (MMU — Memory Management Unit) **транслирует** виртуальные адреса в физические.
- ОС решает, какие страницы памяти реально находятся в ОЗУ, а какие — выгружены на диск.

## Свопинг и страничная адресация

**Свопинг (swap/paging)** — это механизм, при котором ОС может временно выгружать страницы памяти на диск (в файл
подкачки или swap-раздел), освобождая ОЗУ для других процессов. Когда процесс обращается к выгруженной странице,
возникает **страничное прерывание (page fault)**, и ОС загружает страницу обратно.

**Страничная адресация** — организация виртуальной памяти фиксированными блоками (обычно 4 КБ). Каждый виртуальный адрес
разбивается на:

- **номер виртуальной страницы** (старшие биты);
- **смещение внутри страницы** (младшие биты).

MMU использует таблицы страниц (иерархические) для преобразования номера виртуальной страницы в номер физического кадра.

## Планировщик процессов

**Планировщик (scheduler)** — компонент ОС, который решает, какому потоку или процессу отдать процессор в каждый момент
времени. Основные понятия:

- **Параллельность (concurrency)** — несколько задач могут продвигаться одновременно (на макроуровне).
- **Параллелизм (parallelism)** — одновременное выполнение на нескольких ядрах.
- **Квант времени (time slice)** — максимальный интервал, который поток может занимать процессор без переключения.

## Последовательность и кванты времени

На одноядерном процессоре в каждый момент исполняется только один поток. Создание иллюзии одновременной работы
достигается за счёт быстрого переключения между потоками — каждый получает небольшой квант (обычно 10–100 мс).

## Распределение нагрузки между ядрами

На многоядерных системах планировщик распределяет потоки по ядрам, стараясь:

- сохранять баланс загрузки;
- использовать кэш-локальность (держать поток на том же ядре);
- избегать лишних переключений.

## Вытесняющая многозадачность

При **вытесняющей (preemptive) многозадачности** ОС может принудительно снять поток с выполнения по истечении кванта
времени или при появлении более приоритетного потока. Это стандарт для современных ОС (Windows, Linux, macOS).

## Кооперативная многозадачность

В **кооперативной (cooperative) многозадачности** поток или задача самостоятельно решает, когда передать управление.
Передача может происходить явным вызовом (например, `yield`, `await`, `async`), либо при завершении текущей единицы
работы.

**Особенности:**

- Нет принудительных переключений со стороны ОС или планировщика.
- Задача должна добровольно «отпустить» управление, чтобы другие могли выполняться.
- Если задача не отдаёт управление (например, ушла в бесконечный цикл или выполняет тяжёлую синхронную работу), вся
  система может остановиться в рамках данного потока.

**Где используется:**

- **Исторически:** ранние Windows (3.x, 9x), Mac OS Classic. Там процессы/потоки должны были сами вызывать `yield`,
  иначе система зависала.
- **Встроенные системы (bare metal, RTOS)** — часто используются кооперативные планировщики для предсказуемости и
  простоты.
- **Асинхронное программирование — современный стандарт:** любая среда с `async/await` и однопоточным событийным циклом
  по сути реализует кооперативную многозадачность на уровне пользовательских задач (но не на уровне потоков ОС).
  - **JavaScript (Node.js, браузеры)** — асинхронные функции отдают управление событийному циклу на каждом `await`.
    Пока не встретится `await`, задача блокирует выполнение.
  - **Rust (Tokio, async-std)** — `await` передаёт управление рантайму, позволяя выполняться другим задачам.
  - **Python (asyncio)** — `await` переключает на другие сопрограммы.
  - **C# (async/await), Kotlin (coroutines), Swift (async/await), Zig (async)** — везде используется кооперативная
    модель на уровне языка/рантайма.

В этих средах задачи (корутины, футуры, промисы) работают внутри одного или нескольких потоков ОС, но переключения между
ними происходят только в точках `await` или явной передачи управления. Это даёт эффективность и предсказуемость, не
требуя дорогих переключений контекста ядра.

**Важное различие:** кооперативная модель в асинхронном программировании не делает всю систему неотзывчивой — она
блокирует только текущий поток задач. Но если в этом потоке запустить синхронный бесконечный цикл, все сопрограммы этого
потока остановятся.

## Предварительные итоги по процессам и потокам

- **Процесс** = владелец ресурсов (память, файлы, права).
- **Поток** = единица выполнения внутри процесса.
- **Виртуальная память** = изоляция и абстракция физической памяти.
- **Страничная адресация** = способ организации виртуальной памяти.
- **Планировщик** = распределяет время процессора.
- **Вытесняющая многозадачность** = управление от ОС, кооперативная = от самой программы.

## Управление памятью в языках программирования

Переходим от того, как ОС управляет памятью, к тому, как это делают языки программирования и их рантаймы.

### Обзор разных моделей

Модели управления памятью можно классифицировать по способу освобождения:

1. **Ручное управление** — программист сам вызывает `malloc`/`free` или `new`/`delete`.
2. **Автоматическое (сборщики мусора, GC)** — рантайм сам определяет, какие объекты больше не нужны.
3. **Счётчики ссылок** — разновидность автоматического, но с немедленным освобождением.
4. **Статическое / автоматическое (стек)** — время жизни привязано к области видимости.
5. **RAII** — управление ресурсами через конструкторы и деструкторы (C++, Rust).

### Режим реальных адресов (DOS) vs Защищённый режим

- **Реальный режим (real mode)** — использовался в DOS. Процессор работает с физическими адресами, защиты памяти нет.
  Любая программа может писать куда угодно, включая область ОС. Это приводило к частым зависаниям.
- **Защищённый режим (protected mode)** — появился в 80286, полноценно с 80386. Включает страничную адресацию,
  сегментную защиту, разные уровни привилегий (кольца). Современные ОС работают только в защищённом режиме.

### Ручное управление

Языки: C, C++ (частично), Zig, ассемблер.

**Пример на C:**

```c
#include <stdlib.h>

int* create_array(size_t n) {
    int* arr = (int*)malloc(n * sizeof(int));
    if (!arr) return NULL;
    return arr;
    // кто вызвал create_array, тот должен вызвать free()
}

int main() {
    int* data = create_array(100);
    // ... работа с data
    free(data); // явное освобождение
    return 0;
}
```

**Плюсы:** максимальная производительность, контроль, отсутствие неожиданных пауз.

**Минусы:** ошибки утечек памяти, двойного освобождения, use-after-free.

### Счётчики ссылок

Языки: Swift (ARC), Python (с дополнительным сборщиком циклов), Objective-C (ARC), ранний PHP.

Каждый объект хранит число активных ссылок на него. Когда счётчик падает до нуля, объект удаляется немедленно.

**Плюсы:** предсказуемое время освобождения, нет длительных пауз.

**Минусы:**

- оверхед на инкремент/декремент при каждом присваивании;
- циклические ссылки не обнуляются автоматически (требуются слабые ссылки или отдельный сборщик циклов).

**Пример циклической ссылки в Python:**

```python
class Node:
    def __init__(self):
        self.ref = None

a = Node()
b = Node()
a.ref = b
b.ref = a
del a
del b # счётчики не нулевые — утечка (если бы не сборщик циклов)
```

### Трассирующие сборщики мусора (GC)

Языки: Java, C#, Go, JavaScript, Ruby, OCaml.

Сборщик периодически:

- **маркирует (mark)** — находит все достижимые из корней объекты (глобальные переменные, стек вызовов, регистры);
- **удаляет (sweep)** — освобождает недостижимые объекты;
- опционально **уплотняет (compact)** — перемещает живые объекты, устраняя фрагментацию.

**Плюсы:** не нужен ручной контроль, нет проблем с циклами.

**Минусы:** «стоп-зе мир» (STW, stop-the-world) — длительные паузы, непредсказуемость.

### Поколенческая сборка мусора

Наблюдение: большинство объектов живут очень недолго. Поэтому куча делится на **поколения**:

- **Молодое поколение (young generation)** — объекты только что созданы. Собирается часто, быстро (обычно алгоритмом
  копирования).
- **Старое поколение (old generation)** — объекты пережили несколько сборок. Собирается редко.

В V8 (JavaScript) есть также **среднее поколение** (mid generation) в некоторых версиях.

**Эффект:** паузы короткие, потому что в молодом поколении мало живых объектов.

### Инкрементальная сборка

Вместо одного длинного `stop-the-world` инкрементальный сборщик разбивает работу на маленькие порции, чередуя их с
выполнением программы. Это уменьшает максимальную паузу ценой общего замедления.

### Модель копирования для молодого поколения

Один из популярных алгоритмов для молодого поколения — **копирующий сборщик** (часть алгоритма Cheney). Куча делится на
два полупространства:

- `from-space` — активно используется программой.
- `to-space` — резерв.

При сборке:

1. Все достижимые объекты из `from-space` копируются в `to-space` (компактно).
2. Полупространства меняются местами.
3. Старое `from-space` целиком (вместе с мусором) затирается.

Это автоматически устраняет фрагментацию, но требует двойного объёма памяти для молодого поколения.

### Без динамической памяти

В некоторых языках и подходах динамическая память вообще не используется (или используется крайне ограниченно):

- все объекты создаются либо на стеке, либо в статической памяти (на этапе компиляции известны размер и время жизни);
- примеры: некоторые встраиваемые системы, MISRA C, Fortran 77, ранний Pascal без `new`.

**Плюсы:** максимальная предсказуемость, отсутствие переполнения кучи, подходит для жёсткого реального времени.

**Минусы:** невозможность создавать структуры переменного размера без статического верхнего предела.

### Статическая модель

Под «статической моделью» в контексте современных языков чаще всего понимают подход Rust, а также связанные с ним идеи (
например, циклический анализатор ссылок в Swift). Ключевая идея: **компилятор статически (на этапе компиляции)
доказывает корректность управления памятью**, без использования сборщика мусора и без необходимости вручную вызывать
`malloc`/`free`.

**Rust и его модель владения:**

- У каждого значения есть **единственный владелец** (owner).
- Когда владелец выходит из области видимости, память автоматически освобождается (вызывается деструктор, аналог RAII в
  C++).
- Передача значения другому владельцу (перемещение) запрещает доступ из исходного места.
- **Заимствование (borrowing)** — ссылки (`&T` и `&mut T`) с проверкой времён жизни (lifetimes) компилятором.
- Компилятор проверяет, что ссылки не переживают данные, на которые указывают (нет висячих ссылок).

```rust
fn example() {
let s = String::from("hello"); // выделена память в куче
let t = s;                     // владение перемещено в t, s больше недействительна
// println!("{}", s);          // ошибка компиляции!
println!("{}", t);             // OK
} // здесь t выходит из области видимости, память освобождена
```

**Что можно размещать:**

- **На стеке** — значения с известным на этапе компиляции размером (простые структуры, кортежи, массивы фиксированной
  длины, ссылки).
- **В куче** — динамические структуры (`Box<T>`, `Vec<T>`, `String`, `Rc<T>`, `Arc<T>`), но компилятор отслеживает время
  жизни и освобождение.

**Сравнение с другими подходами:**

| Подход             | Освобождение                   | Проверка                                                        | Примеры языков             |
|--------------------|--------------------------------|-----------------------------------------------------------------|----------------------------|
| Ручное             | `free`/`delete`                | Ручная (или анализаторы)                                        | C, C++                     |
| RAII               | деструктор                     | Ручная (язык гарантирует вызов, но не защиту от висячих ссылок) | C++                        |
| Статическая (Rust) | деструктор + проверка владения | Компилятор (строгая, гарантированная)                           | Rust                       |
| Счётчики ссылок    | немедленно при падении до нуля | Рантайм                                                         | Swift (ARC), Python (+ GC) |
| Сборщик мусора     | периодически                   | Рантайм                                                         | Java, C#, Go, JS           |

**Преимущества статической модели Rust:**

- **Безопасность памяти** — никаких утечек, двойных освобождений, use-after-free, гонок данных по памяти (data races) на
  этапе компиляции.
- **Производительность** — никакого сборщика мусора (нет пауз), никакого оверхеда на счётчики ссылок (кроме явных `Rc`/
  `Arc`).
- **Предсказуемость** — освобождение происходит в известные моменты (конец области видимости).

**Ограничения:**

- Некоторые паттерны (циклические структуры, графы) трудно выразить без `Rc<RefCell<T>>` или `Weak`, что возвращает
  часть оверхеда.
- Кривая обучения — проверка заимствований и времён жизни требует привыкания.

**Другие языки с элементами статического управления:**

- **C++** — умные указатели (`unique_ptr`, `shared_ptr`) приближаются к модели Rust, но проверки только рантаймовые (
  кроме `unique_ptr` на уровне семантики перемещения, но без гарантий компилятора).
- **Swift** — ARC (автоматический подсчёт ссылок) — это рантайм, но с сильной статической оптимизацией.
- **Zig** — ручное управление, но с `defer` и аллокаторами как параметрами, без статической проверки.

**Вывод:** статическая модель (в узком смысле — Rust) — это третий путь между сборкой мусора и ручным управлением: *
*безопасность памяти без GC**.

### RAII (Resource Acquisition Is Initialization)

RAII — идиома управления ресурсами, ключевая для C++ и Rust (там похожий механизм называется «владение»). Идея:

- ресурс (память, файл, мьютекс) захватывается в конструкторе;
- освобождается в деструкторе;
- деструктор вызывается автоматически при выходе объекта из области видимости.

```cpp
#include <vector>

void example() {
std::vector<int> vec(100); // выделяет память в куче
// ... работа с vec
} // здесь автоматически вызывается деструктор ~vector(), память освобождена
```

RAII даёт предсказуемое освобождение ресурсов без ручного вызова `free`. В Rust аналогичную роль выполняет `Drop` трейт
и система владения с перемещением.

## Управление памятью в JavaScript

### Модель по умолчанию

JavaScript — язык с автоматической сборкой мусора (трассирующий, поколенческий, инкрементальный). Движки (V8,
JavaScriptCore, SpiderMonkey) сами решают, когда запускать сборщик. Программист не управляет освобождением памяти
напрямую.

### Принудительный вызов GC

В ранних версиях JavaScript принудительно запустить сборку мусора было невозможно. Сейчас в Node.js есть флаг для
запуска GC (при запуске с `--expose-gc`), а также в некоторых браузерных окружениях (но не в стандарте).

```javascript
// Node.js с флагом --expose-gc
if (global.gc) {
    global.gc();
    console.log('GC вызван принудительно');
}
```

Использовать принудительный GC на практике почти никогда не нужно. Это полезно только для тестирования и бенчмарков.

### FinalizationRegistry — отслеживание уничтожения объектов

`FinalizationRegistry` позволяет зарегистрировать колбэк, который выполнится **после** того, как объект был собран
сборщиком мусора. Это **не** деструктор и **не** детерминированное освобождение ресурсов.

```javascript
const registry = new FinalizationRegistry((heldValue) => {
    console.log(`Объект с ${heldValue} был уничтожен GC`);
});

let obj = {data: 42};
registry.register(obj, "пользовательские данные");

obj = null; // объект стал недостижим, в какой-то момент GC вызовет колбэк
```

**Важно:** FinalizationRegistry не гарантирует своевременного выполнения. Его нельзя использовать для критических
ресурсов вроде закрытия файлов.

### using и Symbol.dispose (Disposable объекты)

Нововведение в JavaScript (ES2024, доступно в современных движках) — `using` для явного управления ресурсами. Идея
напоминает RAII или `using` в C#.

```javascript
const getResource = () => ({
    [Symbol.dispose]() {
        console.log('Ресурс освобождён');
    },
    doWork() {
        console.log('Работаю');
    }
});

{
    using resource = getResource();
    resource.doWork();
} // здесь автоматически вызовется resource[Symbol.dispose]()
```

`using` не имеет отношения к сборщику мусора — это синтаксический сахар для блочного управления временем жизни. Отлично
подходит для файловых дескрипторов, соединений с БД, таймеров.

### Слабые ссылки

Обычные ссылки в JavaScript **сильные** — они не дают сборщику удалить объект. **Слабые ссылки (WeakRef)** не
препятствуют сборке. При этом доступ к объекту через `WeakRef` может вернуть `undefined`, если объект уже собран.

```javascript
let obj = {name: "important"};
const weakRef = new WeakRef(obj);

obj = null; // объект может быть собран GC

// позже, когда нужно получить объект
const restored = weakRef.deref();
if (restored) {
    console.log(restored.name);
} else {
    console.log("Объект уже уничтожен");
}
```

Слабые ссылки нужны для **кешей с автоматическим вытеснением**, для предотвращения утечек в паттерне «наблюдатель».
Также существуют `WeakMap` и `WeakSet`, где ключи — слабые ссылки.

## Итоги лекции

- **Операционная система** обеспечивает изоляцию процессов и управление ресурсами через **виртуальную память**, *
  *страничную адресацию** и **планировщик**.
- **Вытесняющая многозадачность** — стандарт для современных ОС; **кооперативная** исторически старше и менее надёжна.
- **Управление памятью в языках программирования** варьируется от полного ручного контроля (C) до полностью
  автоматической сборки мусора (Java, C#, Go, JavaScript).
- **Счётчики ссылок** предсказуемы, но не справляются с циклами (требуют дополнительных механизмов).
- **Поколенческий GC** использует наблюдение, что большинство объектов живёт недолго, что даёт короткие паузы.
- **RAII** в C++ и владение в Rust дают автоматическое, но предсказуемое освобождение без сборщика мусора.
- **JavaScript** — поколенческий трассирующий GC. Новые возможности (`FinalizationRegistry`, `WeakRef`, `using`) дают
  больше контроля над временем жизни, но не превращают JS в язык с ручным управлением.
- Понимание процессов, потоков и моделей памяти помогает писать более эффективные и безопасные программы, особенно на
  стыке языков (например, модули Node.js на C++ или WebAssembly).
