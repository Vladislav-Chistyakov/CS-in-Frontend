# AI конспект для лекции #14

## Что такое алгоритмическая сложность

**Алгоритмическая сложность** — это характеристика того, как быстро растёт время выполнения (или потребление памяти)
алгоритма при увеличении размера входных данных. Она обозначается **О-нотацией** (асимптотическая сложность).

Основные классы сложности (от лучшей к худшей):

| Обозначение | Название                | Пример                                   |
|-------------|-------------------------|------------------------------------------|
| O(1)        | Константная             | Доступ к элементу массива по индексу     |
| O(log n)    | Логарифмическая         | Бинарный поиск в отсортированном массиве |
| O(n)        | Линейная                | Поиск элемента в несортированном массиве |
| O(n log n)  | Линейно-логарифмическая | Быстрая сортировка, сортировка слиянием  |
| O(n²)       | Квадратичная            | Пузырьковая сортировка                   |
| O(2ⁿ)       | Экспоненциальная        | Задача коммивояжёра (полный перебор)     |

## Анализ операций структур данных: вставка, удаление, поиск

Разные структуры данных имеют разные сложности для базовых операций:

| Структура данных                 | Вставка     | Удаление    | Поиск по значению | Доступ по индексу |
|----------------------------------|-------------|-------------|-------------------|-------------------|
| Массив (несортированный)         | O(1)*       | O(n)**      | O(n)              | O(1)              |
| Массив (сортированный)           | O(n)        | O(n)        | O(log n)          | O(1)              |
| Связный список                   | O(1)***     | O(1)***     | O(n)              | O(n)              |
| Хэш-таблица                      | O(1) средн. | O(1) средн. | O(1) средн.       | —                 |
| Дерево поиска (сбалансированное) | O(log n)    | O(log n)    | O(log n)          | —                 |

> *Вставка в конец массива с амортизированным O(1). Вставка в середину — O(n).  
> **Удаление из конца — O(1), из середины — O(n).  
> ***Вставка/удаление в известную позицию (например, после узла, на который есть ссылка) — O(1). Поиск позиции — O(n).

## Влияние кэша и локальности данных

О-нотация не учитывает реальную скорость работы с памятью. **Локальность данных** (сколько данных подряд используется в
памяти) критически важна:

- **Массивы** — элементы хранятся непрерывно. При переборе они подгружаются в кэш процессора блоками (cache line), что
  ускоряет доступ в десятки раз по сравнению со случайными обращениями.
- **Связные списки** — узлы разбросаны по куче. Перебор порождает много промахов кэша (cache misses), делая реальное
  время работы значительно хуже, чем предсказывает O-нотация.
- **Хэш-таблицы** — средняя локальность, но при открытой адресации лучше, чем при цепочках.

**Вывод:** O(1) хэш-таблицы или связного списка на практике может быть медленнее O(log n) бинарного поиска в массиве,
если последний полностью помещается в кэш.

## Параллелизм может сделать алгоритм с худшей сложностью быстрее

О-нотация предполагает последовательное выполнение. При распараллеливании ситуация меняется:

- Пузырьковая сортировка (O(n²)) на 1000 ядрах может обогнать быструю сортировку (O(n log n)) на одном ядре для
  достаточно больших данных.
- Поиск в несортированном массиве (O(n)) на 16 ядрах линейно ускоряется до O(n / p).

Однако не все алгоритмы хорошо параллелятся — учитывайте накладные расходы на синхронизацию и передачу данных.

## Тип памяти (HDD, сеть) меняют правила

Сложность, измеряемая в количестве операций, плохо работает, когда сами операции имеют разную стоимость:

| Тип памяти/доступа       | Относительная стоимость | Последствия для выбора алгоритма                                            |
|--------------------------|-------------------------|-----------------------------------------------------------------------------|
| Регистры CPU             | 1 (база)                | —                                                                           |
| Кэш L1/L2/L3             | ~2-30                   | Массивы (локальность) выигрывают                                            |
| Оперативная память (RAM) | ~100-300                | Обращения к ОЗУ дороги, но предсказуемы                                     |
| Твёрдый диск (SSD)       | ~10 000-100 000         | Минимизируйте число обращений, используйте последовательное чтение          |
| Твёрдый диск (HDD)       | ~1 000 000-10 000 000   | Последовательное чтение значительно быстрее случайного                      |
| Сеть (удалённый сервер)  | ~10 000 000-100 000 000 | Алгоритм, уменьшающий число запросов (даже с худшей O-нотацией), выигрывает |

**Пример:** поиск в несортированном массиве на HDD через бинарный поиск невозможен (нет произвольного доступа), а полный
перебор требует чтения всего файла последовательно — это может быть быстрее, чем пытаться организовать индексы.

## Что такое коллекции

**Коллекции** — это структуры данных, предназначенные для хранения и управления группами объектов. Основные типы
коллекций:

- **Последовательные** — порядок элементов важен (массив, список, стек, очередь).
- **Ассоциативные** — доступ по ключу (словарь, хэш-таблица, Map).
- **Множества** — уникальные элементы (Set).
- **Древовидные** — иерархические отношения (деревья поиска, кучи).

## Связанный список

**Связный список** — структура, где каждый элемент (узел) хранит значение и ссылку на следующий (и, возможно,
предыдущий) узел.

**Преимущества:**

- Вставка и удаление в известную позицию — O(1).
- Динамический размер (не требует переаллокации, как массив).
- Не требует непрерывного блока памяти.

**Недостатки:**

- Нет произвольного доступа (чтобы найти элемент, нужно пройти с начала).
- Промахи кэша (узлы разбросаны по памяти).
- Дополнительная память под ссылки.

## Хэш-таблица

Хэш-таблица — структура, отображающая ключи на значения через хэш-функцию. Подробно рассмотрена в предыдущей лекции.

**Ключевые характеристики:**

- Вставка, удаление, поиск — O(1) в среднем.
- Не упорядочена (порядок ключей не гарантирован, кроме специальных реализаций).
- Требует хорошей хэш-функции.

## Массив

**Массив** — непрерывная область памяти с элементами одинакового размера. Доступ по индексу — O(1).

**Преимущества:**

- Быстрый доступ по индексу.
- Отличная локальность данных (кэш-дружелюбен).
- Минимальные накладные расходы (только данные).

**Недостатки:**

- Вставка и удаление из середины — O(n) (нужно сдвигать элементы).
- Фиксированный размер (в статических языках) или амортизированное расширение (в динамических).

### Удаление из массива за O(1)

Классическое удаление из середины массива — O(n), так как элементы сдвигаются. Но есть трюки для получения O(1) в
некоторых сценариях:

1. **Удаление без сохранения порядка** (нестабильное удаление) — заменить удаляемый элемент последним, затем уменьшить
   размер на 1. Работает, если порядок элементов не важен.

    ```javascript
    // Нестабильное удаление
    function deleteUnordered(arr, index) {
        arr[index] = arr[arr.length - 1];
        arr.pop();
    }
    
    // Порядок элементов нарушен, но удаление за O(1)
    ```

2. **Битовые флаги** — пометить элемент как удалённый, реально удалять при сборке мусора / перестроении.

3. **Черновик (sparse)** — использовать разреженное представление.

### Бинарный поиск: деление пополам

**Бинарный поиск** работает только на **отсортированном массиве**. Идея: на каждом шаге делим область поиска пополам.

```javascript
function binarySearch(arr, target) {
    let left = 0;
    let right = arr.length - 1;

    while (left <= right) {
        const mid = Math.floor((left + right) / 2);
        if (arr[mid] === target) return mid;
        if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1; // не найден

}
```

**Сложность:** O(log n).

**Важное замечание о вычислении середины:** в некоторых языках `(left + right) / 2` может переполниться при больших
`left` и `right`. Решение: `left + (right - left) / 2`.

**Применение:**

- Поиск в отсортированных данных.
- Нахождение границ (первый/последний элемент, удовлетворяющий условию).
- Оптимизация «угадай число».
- В качестве подзадачи в более сложных алгоритмах.

## Вставка и удаление в отсортированном массиве

Вставка и удаление в отсортированном массиве требуют:

1. Поиск позиции (O(log n) через бинарный поиск).
2. Сдвиг элементов (O(n) для вставки или удаления).

Итоговая сложность — **O(n)**. Если операции вставки/удаления частые, отсортированный массив подходит плохо —
используйте сбалансированные деревья или хэш-таблицы.

## Сортировка

### Пузырьковая сортировка (Bubble sort)

Простейший алгоритм: многократно проходим по массиву, меняя соседние элементы, если они стоят в неправильном порядке.

**Сложность:** O(n²) в худшем и среднем случае, O(n) в лучшем (если массив уже отсортирован).

**Плюсы:** прост в реализации.
**Минусы:** очень медленный для больших массивов.

```javascript
function bubbleSort(arr) {
    for (let i = 0; i < arr.length - 1; i++) {
        for (let j = 0; j < arr.length - 1 - i; j++) {
            if (arr[j] > arr[j + 1]) {
                [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
            }
        }
    }
    return arr;
}
```

### Быстрая сортировка (Quick sort)

Рекурсивный алгоритм: выбирается опорный элемент (pivot), массив разбивается на элементы меньше и больше опорного, затем
каждая часть сортируется рекурсивно.

**Сложность:** O(n log n) в среднем, O(n²) в худшем (при неудачном выборе опорного элемента).

**Особенности:**

- Сортировка на месте (in-place), мало дополнительной памяти.
- Очень быстра на практике (отсюда название).
- Нестабильная (не сохраняет порядок равных элементов).

### Сортировка слиянием (Merge sort)

Рекурсивно делит массив на две половины, сортирует каждую, затем сливает (merge) две отсортированные половины.

**Сложность:** O(n log n) во всех случаях.

**Особенности:**

- Требует O(n) дополнительной памяти.
- Стабильная (сохраняет порядок равных элементов).
- Хороша для связных списков и внешней сортировки (работа с диском).

### Сортировка кучей (Heap sort)

Преобразует массив в кучу (heap), затем многократно извлекает корневой элемент.

**Сложность:** O(n log n) во всех случаях.

**Особенности:**

- Сортировка на месте (O(1) дополнительной памяти).
- Нестабильная.
- На практике медленнее быстрой сортировки, но гарантирует O(n log n) без худших случаев.

### Поразрядная сортировка (Radix sort)

Сортирует числа, обрабатывая разряды по отдельности (сначала младший разряд, затем старший).

**Сложность:** O(k × n), где k — количество разрядов.

**Особенности:**

- Работает только для целых чисел (или строк фиксированной длины).
- Может быть быстрее сравнений для больших n и маленького k.
- Требует дополнительной памяти (обычно O(n)).

### Стабильность (устойчивость) сортировки

**Стабильная сортировка** сохраняет относительный порядок равных элементов.

Пример: сортировка людей по возрасту, а затем по имени — стабильность сохранит возрастной порядок внутри имени.

| Алгоритм    | Стабильность          |
|-------------|-----------------------|
| Пузырьковая | стабильна             |
| Вставками   | стабильна             |
| Слиянием    | стабильна             |
| Быстрая     | обычно нестабильна    |
| Кучей       | нестабильна           |
| Поразрядная | зависит от реализации |

### Сортировка в JavaScript

В JavaScript сортировка массивов выполняется методом `arr.sort([compareFunction])`.

**Особенности:**

- Алгоритм зависит от движка:
  - V8 (Chrome, Node.js) — TimSort (гибрид слияния и вставок) для больших массивов, быстрая сортировка для маленьких (
    исторически).
  - SpiderMonkey (Firefox) — TimSort.
  - JavaScriptCore (Safari) — слиянием или сортировка на основе блочного слияния.
- По умолчанию элементы преобразуются в строки и сортируются лексикографически (это часто приводит к неожиданностям).

```javascript
const nums = [10, 5, 2, 20];
nums.sort(); // [10, 2, 20, 5] — лексикографическая сортировка!

// Правильная сортировка чисел
nums.sort((a, b) => a - b); // [2, 5, 10, 20]
```

- `sort()` сортирует на месте (мутирует исходный массив) и возвращает ссылку на него.
- Сложность гарантируется O(n log n) в современных движках.

## Деревья поиска

**Дерево поиска** — структура данных, где каждый узел имеет не более двух детей, и для каждого узла выполняется условие:
все значения в левом поддереве меньше значения узла, все значения в правом — больше.

### Двоичное дерево поиска (BST)

```javascript
class BSTNode {
    constructor(value) {
        this.value = value;
        this.left = null;
        this.right = null;
    }
}

class BinarySearchTree {
    constructor() {
        this.root = null;
    }

    insert(value) {
        const node = new BSTNode(value);
        if (!this.root) {
            this.root = node;
            return;
        }

        let current = this.root;
        while (true) {
            if (value < current.value) {
                if (!current.left) {
                    current.left = node;
                    return;
                }
                current = current.left;
            } else {
                if (!current.right) {
                    current.right = node;
                    return;
                }
                current = current.right;
            }
        }
    }

    search(value) {
        let current = this.root;
        while (current) {
            if (value === current.value) return current;
            if (value < current.value) current = current.left;
            else current = current.right;
        }
        return null;
    }

}
```

**Сложность:**

- В среднем — O(log n) для вставки, поиска, удаления.
- В худшем случае (дерево вырождается в связный список) — O(n).

**Решение — сбалансированные деревья** (AVL, красно-чёрное дерево), которые гарантируют O(log n) в худшем случае.
Реализация значительно сложнее.

**Применение:**

- Словари и множества (например, `std::map` в C++ — красно-чёрное дерево).
- Индексы в базах данных.
- Вычисление порядковых статистик.

## Итоги лекции

- **Алгоритмическая сложность** (О-нотация) — абстрактная мера роста времени/памяти, но **не учитывает реальные факторы
  **: кэш, локальность, параллелизм, тип памяти (HDD/SSD/сеть).
- **Массивы** — лучшая локальность, быстрый доступ по индексу, но медленная вставка/удаление из середины. Бинарный
  поиск (O(log n)) работает только на отсортированных массивах.
- **Связные списки** — быстрая вставка/удаление в известной позиции, но медленный поиск и плохая локальность.
- **Хэш-таблицы** — O(1) в среднем для всех операций, но неупорядочены и имеют накладные расходы.
- **Деревья поиска** — O(log n) для основных операций, упорядочены, но сложнее в реализации.
- **Сортировки** — от O(n²) (пузырьковая) до O(n log n) (быстрая, слиянием, кучей). Выбор зависит от размера данных,
  доступной памяти, необходимости стабильности.
- **Стабильность сортировки** — сохранение порядка равных элементов, важна для многоуровневой сортировки.
- **JavaScript** — `sort()` с компаратором, гарантия O(n log n) в современных движках.
