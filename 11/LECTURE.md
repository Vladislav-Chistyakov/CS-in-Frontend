# AI конспект для лекции #11

## Введение: память как ресурс

Программа, когда запускается, получает адресное пространство — по сути, большой непрерывный массив байтов, адреса
которого идут от `0` до `N-1`. Разным компонентам программы нужен доступ к этой памяти, но с разными требованиями:

- одним нужна максимальная скорость;
- другим — строгая дисциплина «кто создал, тот и удалил»;
- третьим — гибкость, чтобы данные жили дольше, чем породившая их функция.

Чтобы удовлетворить всем, память разбивается на **логические сегменты**. Каждый сегмент работает по своим правилам: кто
отвечает за выделение, кто за освобождение, как долго там живут данные.

Основные сегменты памяти:

1. **Сегмент кода** — машинные инструкции, обычно только для чтения и исполнения.
2. **Сегмент данных** — глобальные и статические переменные.
3. **Стек** — автоматическая память для вызовов функций.
4. **Куча** — динамическая память с ручным или автоматическим управлением.

## Стек

### Что такое стек как структура данных

Стек — это абстрактная структура данных, работающая по принципу **LIFO** (Last In, First Out): последним пришёл, первым
ушёл. У него есть две базовые операции:

- `push` — поместить элемент на вершину;
- `pop` — удалить элемент с вершины.

### Сегмент стека как фиксированный массив

В подавляющем большинстве языков и сред исполнения **сегмент стека реализован как обычный фиксированный массив** в
виртуальной памяти процесса. Это сделано по нескольким причинам:

- **Простота** — массив — самая простая структура данных, которую можно реализовать парой регистров (указатель начала
  стека и указатель вершины).
- **Эффективность** — выделение памяти под локальную переменную или вызов функции сводится к одной арифметической
  операции: `stack_pointer -= размер`. Освобождение — к обратному сдвигу. Никакого поиска свободных блоков, как в куче.
- **Предсказуемость** — операции со стеком занимают константное и очень маленькое время.

**Размер стека фиксирован.** Он задаётся при создании потока исполнения (обычно 1–8 МБ) и не меняется в процессе работы.
Почему нельзя динамически расширять стек?

- Расширение потребовало бы выделения нового массива и **копирования всех текущих стековых данных** в новую область. Это
  дорого и разрушает локальность ссылок.
- Стек используется настолько интенсивно (каждый вызов функции, каждое возвращение), что накладные расходы на проверку
  необходимости расширения и потенциальное копирование были бы неприемлемы.
- Нарушается атомарность базовых операций — `push` перестаёт быть простым сдвигом указателя.

Поэтому стек остаётся фиксированным. При переполнении программа получает ошибку **Stack overflow**.

### Стековый кадр (frame)

Каждый вызов функции порождает на стеке **стековый кадр** (stack frame). В кадре хранятся:

- аргументы, переданные в функцию;
- локальные переменные;
- адрес возврата — место в коде, куда нужно перейти после завершения функции;
- сохранённое состояние регистров (чтобы восстановить их после возврата).

**Пример на условном ассемблере (схематично):**

```asm
Вызов функции A:
push адрес_возврата
push значение_аргумента1
...
выделение места под локальные переменные (сдвиг SP)

Внутри A: доступ к локальным переменным через смещения от SP или BP
```

### Пролог и эпилог функции

Перед выполнением тела функции компилятор вставляет машинные инструкции **пролога**:

- сохранить текущее значение базового регистра (если используется);
- скопировать текущий указатель стека в базовый регистр (чтобы иметь фиксированную точку отсчёта внутри кадра);
- сдвинуть указатель стека, чтобы выделить место под локальные переменные.

После завершения функции выполняется **эпилог**:

- сдвинуть указатель стека обратно (освободить локальные переменные);
- восстановить сохранённые регистры;
- перейти по адресу возврата.

### Рекурсия и переполнение стека

Рекурсия — это вызов функции из самой себя. Каждый новый рекурсивный вызов создаёт новый стековый кадр. Глубина рекурсии
ограничена размером стекового сегмента.

**Пример на JavaScript:**

```javascript
function factorial(n) {
    if (n === 1) return 1;
    return n * factorial(n - 1);
}

factorial(100000); // => Stack overflow
```

При рекурсивном вызове `factorial(100000)` стек занял бы 100 000 кадров. При размере кадра, скажем, 32 байта, это 3.2
МБ — уложиться можно, но многие среды ограничивают стек агрессивнее. Главное: фиксированный размер стека означает, что
всегда есть максимальная глубина вызовов, которую нельзя превысить.

### Копирование возвращаемых значений

В большинстве языков возвращаемое значение копируется из кадра вызываемой функции в кадр вызывающей. Для маленьких
данных (числа, короткие структуры) это быстро. Однако если функция возвращает большой объект на стеке, начинаются
проблемы:

- тратится время на копирование байтов;
- растёт расход памяти на стеке (объект существует одновременно в двух кадрах в момент возврата).

Поэтому крупные данные предпочитают передавать через кучу.

## Куча

### Почему стека недостаточно

У стека есть фундаментальные ограничения, вытекающие из его реализации как фиксированного массива:

- Размер стека фиксирован и относительно невелик (МБ, а не ГБ).
- Данные живут ровно столько, сколько выполняется функция (время жизни автоматическое и строго вложенное).
- Нельзя создать объект в одной функции и вернуть его так, чтобы он пережил завершение этой функции, — при возврате он
  либо скопируется, либо будет уничтожен.

Для задач, где время жизни данных не совпадает со временем жизни стекового кадра, нужна **куча** (heap).

### Что такое куча

Куча — это сегмент памяти, выделение в котором происходит не автоматически, а по явному запросу (или под управлением
сборщика мусора). Данные в куче живут до тех пор, пока их специально не освободят (явно или автоматически).

**В C (ручное управление):**

```c
int* p = (int*)malloc(sizeof(int));
*p = 42;
free(p);  // явное освобождение
```

**В JavaScript (автоматическое управление):**

```javascript
let obj = {value: 42};  // объект создан в куче
// когда на obj не останется ссылок, память освободит сборщик мусора
```

### Как организована куча

В отличие от стека, куча не является ни стеком, ни массивом с дисциплиной LIFO. Это **пул памяти произвольного доступа**, которым управляет **аллокатор**. Аллокатор поддерживает структуры данных (обычно связные списки свободных блоков или
двоичные деревья), чтобы:

- находить свободный блок под запрошенный размер;
- отмечать блок занятым;
- при освобождении объединять соседние свободные блоки (дефрагментация).

### Фрагментация

Со временем в куче возникает **фрагментация**:

- **Внешняя** — свободные блоки разбросаны по куче, нет одного непрерывного блока нужного размера, хотя суммарно
  свободной памяти достаточно.
- **Внутренняя** — аллокатор выделил блок чуть больше запрошенного (например, из-за выравнивания), и часть памяти
  теряется внутри блока.

Фрагментация — одна из главных проблем ручного управления кучей. Автоматические сборщики мусора часто включают сжатие (compaction) — перемещение живых объектов в начало кучи, чтобы устранить внешнюю фрагментацию.

### Преимущества кучи

- Данные живут независимо от функций и их вызовов.
- Можно создавать динамические структуры (связные списки, хеш-таблицы, деревья), размер которых заранее неизвестен.
- Размер кучи ограничен только доступной памятью (или настройками процесса).

### Сборщик мусора (GC)

В языках с автоматическим управлением памятью (JavaScript, Java, C#, Go, Python) работает **сборщик мусора**. Он
периодически:

1. Находит все объекты в куче, достижимые из корней (глобальные переменные, текущий стек, регистры).
2. Освобождает недостижимые объекты (они больше никогда не могут быть использованы).
3. Опционально выполняет сжатие, чтобы уменьшить фрагментацию.

**Утечки памяти** возникают, когда программист случайно сохраняет ссылку на объект, который больше не нужен (например,
забыл отписаться от события). Сборщик видит, что объект достижим, и не удаляет его, хотя по логике программы он уже не
нужен.

### Счётчики ссылок

Альтернатива полноценному сборщику мусора — **счётчики ссылок**. Каждый объект хранит счётчик, сколько ссылок на него
существует. Когда счётчик становится равным нулю, память освобождается сразу.

Проблема: циклические ссылки (объект `A` ссылается на `B`, а `B` на `A`) — счётчики никогда не обнуляются. Требуется
дополнительный механизм (обычно слабые ссылки или периодический сборщик циклов).

### Что быстрее: стек или куча?

**Стек значительно быстрее**:

- Выделение памяти на стеке — это одна арифметическая операция: `SP -= размер`. Никакого поиска свободных блоков,
  никаких блокировок.
- Освобождение — обратный сдвиг.
- Стек обладает отличной **пространственной локальностью** — данные почти всегда лежат в кэше процессора.

**Куча медленнее**:

- Аллокатор должен найти свободный блок подходящего размера (потенциально обход связного списка или дерева).
- В многопоточных средах требуется синхронизация (например, мьютекс в аллокаторе, если куча общая).
- Автоматические сборщики вносят непредсказуемые паузы на сборку мусора.

## Особенности JavaScript

### Формальная модель: всё — ссылки (значения-указатели)

В спецификации JavaScript и в большинстве движков (V8, SpiderMonkey, JavaScriptCore) используется единая модель: **все
значения — это ссылки (указатели) фиксированного размера** (обычно 64 или 32 бита). Когда вы пишете:

```javascript
let x = 42;
let y = {value: 42};
```

Переменная `x` хранит ссылку, которая указывает на значение `42`. Переменная `y` хранит ссылку, которая указывает на
объект в куче. Разница в том, **куда указывает эта ссылка** и как интерпретируются биты.

### Техника «тегированных указателей» и SMI (Small Integers)

Чтобы не выделять память в куче для каждого маленького числа, движки используют оптимизацию: **ссылка сама может
содержать данные**, а не адрес. Это называется **тегирование указателей** (pointer tagging).

В V8 (движок Chrome и Node.js) используется следующий подход:

- **SMI (Small Integer)** — 31-битное целое (на 64-битных системах) хранится прямо внутри ссылки. Последний бит (или
  несколько) ссылки отводится под тег, который говорит: «это не адрес, а сразу значение».
- Если целое выходит за пределы SMI-диапазона, оно «боксируется» (boxed) — для него создаётся объект-обёртка в куче, и
  ссылка становится обычным указателем на этот объект.

**Пример:**

```javascript
let a = 42;        // SMI — значение хранится прямо в ссылке
let b = 2 ** 40;   // большое число — уже объект в куче (обычно HeapNumber)
```

Аналогичные техники применяются к:

- **логическим значениям** (`true`/`false`) — специальные глобальные объекты, на которые указывают тегированные ссылки;
- **`null`** и **`undefined`** — тоже реализованы как специальные значения ссылок;
- **маленьким строкам** — строки до определённой длины могут храниться внутри самой ссылки или в специальном пуле.

### Где физически находятся данные

С учётом вышесказанного:

- **SMI-числа, `null`, `undefined`, `true`, `false`** — хранятся прямо в ссылке (на стеке). Никакого дополнительного
  выделения памяти.
- **Объекты, массивы, функции, большие числа (HeapNumber), длинные строки** — ссылка на стеке указывает на блок в куче,
  где хранятся реальные данные.
- **Замыкания** — переменные, захваченные замыканием, размещаются в куче, потому что должны пережить выполнение внешней
  функции.

### Передача значений в функции

Поскольку все значения — это ссылки, формально **в JavaScript всегда передаётся ссылка** (копия ссылки). Однако эффект
для программиста различается в зависимости от того, **можно ли изменить целевые данные через эту ссылку**:

- Для SMI-чисел, `null`, `undefined`, булевых значений — данных, хранящихся прямо в ссылке, — изменить их нельзя.
  Поэтому поведение выглядит как «передача по значению».
- Для объектов, массивов — через копию ссылки можно изменить сам объект (но не переменную, хранящую оригинальную
  ссылку). Это выглядит как «передача по ссылке».

**Примеры:**

```javascript
// SMI — нельзя изменить оригинал
let a = 10;

function change(x) {
    x = 20; // Меняем локальную копию ссылки, но не оригинал
}

change(a);
console.log(a); // 10

// Объект — можно изменить сам объект
let obj = {value: 10};

function change(o) {
    o.value = 20; // Меняем объект через копию ссылки
}

change(obj);
console.log(obj.value); // 20

// Перезапись ссылки не влияет на оригинал
let obj2 = {value: 10};

function reassign(o) {
    o = {value: 20}; // Меняем локальную копию ссылки
}

reassign(obj2);
console.log(obj2.value); // 10 — объект не изменился
```

### Замыкания и куча

Замыкание — это функция, которая «запоминает» переменные из внешней функции, даже после того, как та завершилась.

```javascript
function createCounter() {
    let count = 0;
    return function () {
        count++;
        return count;
    };
}

const counter = createCounter();
```

Переменная `count` должна жить дольше, чем `createCounter`. Если бы `count` лежала на стеке, она была бы уничтожена при
выходе из `createCounter`. Поэтому движок JavaScript размещает `count` **в куче**, а замыкание получает ссылку на неё.

### Оптимизации движков

Современные движки (V8, SpiderMonkey, JavaScriptCore) активно оптимизируют управление памятью:

- **Escape analysis (анализ ухода)** — если движок может доказать, что объект не переживает функцию и не передаётся
  наружу, он может разместить его на стеке, а не в куче (даже если формально это объект).
- **Hidden classes** — объекты с одинаковыми свойствами получают скрытый класс, экономя память на хранении описания
  свойств.
- **Generational GC** — куча делится на поколения. Молодые объекты собираются часто (быстро), старые — редко (не мешают
  производительности).
- **Inline caching** — запоминает результаты поиска свойств объектов, ускоряя последующие доступы.

### Зачем знать это фронтенд-разработчику

Понимание реальной модели памяти JavaScript помогает:

- избегать утечек памяти (неочищенные таймеры, забытые обработчики событий, ссылки в замыканиях);
- понимать, почему `Maximum call stack size exceeded` — это ошибка переполнения фиксированного стека из-за слишком
  глубокой рекурсии или вложенных вызовов;
- осознавать, что передача больших объектов не вызывает их копирования (копируется только ссылка фиксированного
  размера);
- писать код, который дружественен сборщику мусора (обнулять ссылки на крупные объекты, когда они больше не нужны);
- понимать ограничения SMI и не удивляться, почему очень большие числа работают медленнее.

## Другие сегменты памяти

Помимо стека и кучи, в памяти процесса есть и другие сегменты:

1. **Сегмент кода (text)** — машинные инструкции. Обычно защищён от записи. Размер фиксирован после загрузки программы.
2. **Сегмент данных (data)** — инициализированные глобальные и статические переменные.
3. **Сегмент BSS** — неинициализированные глобальные и статические переменные. В исполняемом файле не занимает места,
   при загрузке обнуляется.
4. **Сегмент литералов (rodata)** — строковые и другие константы, только для чтения.

Стек и куча обычно расположены так, чтобы расти навстречу друг другу: куча — в сторону увеличения адресов, стек — в
сторону уменьшения. Это позволяет максимально гибко использовать доступное пространство.

### Языки без кучи

Некоторые языки и среды обходятся без динамической памяти:

- **Forth** — работает почти исключительно со стеком.
- **Безопасные подмножества C** (например, MISRA C с запретом `malloc`) — часто используются в авиации, медицине,
  автомобильной промышленности. Вся память выделяется статически или на стеке.
- **Микроконтроллеры bare metal** — куча часто отсутствует или сильно ограничена; вся память известна на этапе
  компиляции.

### Регистры процессора

Регистры процессора — это сверхбыстрые ячейки внутри CPU (обычно 16–64 штуки). Доступ к регистру занимает **1 такт**.
Доступ к оперативной памяти — **десятки или сотни тактов** (плюс промахи кэша).

Компиляторы стремятся держать часто используемые локальные переменные в регистрах, а не на стеке. Например, счётчик
цикла `for (let i = 0; i < n; i++)` с большой вероятностью будет жить в регистре, никогда не попадая в стек.

### Настройка размера кучи в Node.js

В Node.js можно управлять размером кучи через флаги:

```bash
node --max-old-space-size=4096 script.js  # 4 ГБ для старой области
node --max-semi-space-size=64 script.js   # 64 МБ для молодой области
```

## Итоги лекции

- **Стек** — фиксированный массив, дисциплина LIFO. Выделение — сдвиг указателя, освобождение — обратный сдвиг. Очень
  быстрый, но маленький и с автоматическим временем жизни.
- **Куча** — пул памяти с произвольным доступом, управляемый аллокатором. Медленнее стека, подвержена фрагментации, но
  позволяет создавать данные с произвольным временем жизни.
- **В JavaScript** все значения — это ссылки фиксированного размера. SMI и некоторые другие значения хранятся прямо в
  ссылке (тегированные указатели). Объекты, массивы, большие числа — в куче.
- **Замыкания** перемещают захваченные переменные в кучу, потому что они должны пережить выполнение внешней функции.
- **Сборщик мусора** автоматически освобождает недостижимые объекты в куче, но может создавать паузы. Понимание его
  работы помогает избегать утечек.
- Фиксированный размер стека объясняет ограниченную глубину рекурсии и важность хвостовой рекурсии (где это
  поддерживается).
- Оптимизации движков (escape analysis, hidden classes, generational GC) могут размещать некоторые объекты на стеке даже
  в JavaScript, но это деталь внутренней реализации.
