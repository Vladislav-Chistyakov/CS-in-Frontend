# AI конспект для лекции #13

## Можно ли сделать индекс в массиве нечисловым?

В классическом массиве индексы — это целые числа (обычно от 0 до N-1). Но в реальных задачах хочется индексировать
данные строковыми ключами: имена пользователей, ID, названия товаров. Для этого нужна **структура данных с ассоциативным
доступом** (ключ → значение).

## Ассоциативный доступ

Ассоциативный доступ означает, что поиск значения происходит не по числовому индексу, а по **ключу** любого типа (
строка, число, объект). Ключ должен быть:

- **сравнимым** (чтобы определить равенство);
- **хэшируемым** (чтобы быстро вычислять позицию в таблице).

## Структуры и кортежи как компромисс

Прежде чем перейти к динамическим ассоциативным массивам, стоит упомянуть **структуры** и **кортежи** — они тоже
предоставляют доступ к данным по имени (или по позиции), но с принципиально иными характеристиками:

**Структуры (struct):**

- Поля фиксированы и известны на этапе компиляции.
- Доступ к полю — это смещение в памяти (константная операция, но не хэширование).
- Нет возможности добавить новое поле во время выполнения.
- Память выделяется непрерывно (обычно на стеке или внутри другого объекта).
- Примеры: `struct` в C, C++, Rust; `record` в C#; `typedef struct` в Go.

**Кортежи (tuple):**

- Доступ по индексу (позиции), а не по имени (хотя некоторые языки добавляют именованные кортежи).
- Размер и типы фиксированы.

```c
// Структура в C — фиксированные поля, доступ по имени
struct Point {
    int x;
    int y;
};

struct Point p = {10, 20};
p.x = 30;  // доступ по смещению, очень быстро
```

```python
// Кортеж в Python — фиксированная длина, доступ по индексу
point = (10, 20)
print(point[0])  // 10
```

**Сравнение с ассоциативным массивом (хэш-таблицей):**

| Характеристика             | Структура / Кортеж              | Ассоциативный массив          |
|----------------------------|---------------------------------|-------------------------------|
| Доступ по имени            | Да (структура)                  | Да                            |
| Добавление полей           | Нет (фиксированы)               | Да                            |
| Удаление полей             | Нет                             | Да                            |
| Тип ключа                  | Только имя поля (идентификатор) | Любой (строка, число, объект) |
| Производительность доступа | O(1) — смещение в памяти        | O(1) среднее (хэш)            |
| Память                     | Компактно                       | Больше (служебная информация) |
| Гибкость                   | Низкая (статическая схема)      | Высокая (динамическая)        |

**Вывод:** структуры и кортежи — выбор для статически известных схем данных, где важна производительность и
компактность. Ассоциативные массивы — для динамических структур, где набор ключей заранее неизвестен и может меняться во
время выполнения.

## Ассоциативные массивы

**Ассоциативный массив (словарь, map)** — абстрактная структура данных, поддерживающая операции:

- `set(key, value)` — добавить или обновить пару.
- `get(key)` — получить значение по ключу.
- `delete(key)` — удалить пару.
- `has(key)` — проверить наличие ключа.

Самая популярная и эффективная реализация ассоциативного массива — **хэш-таблица**.

## Хэш-таблица

**Хэш-таблица** использует массив фиксированного размера и **хэш-функцию**, которая преобразует ключ в индекс массива.

Принцип работы:

1. Вычислить `hash = хэш-функция(ключ)`.
2. Вычислить `индекс = hash % размер_массива`.
3. Поместить значение в ячейку массива с полученным индексом.
4. При поиске — повторить шаги 1-2 и прочитать значение.

## Что такое хэш-функция

**Хэш-функция** — это функция, которая преобразует входные данные (ключ) в целое число (хэш, дайджест). Требования к
хэш-функции:

- **Детерминированность** — одинаковый ключ всегда даёт одинаковый хэш.
- **Равномерность** — хэши распределены по диапазону примерно равномерно (чтобы избежать коллизий).
- **Быстрота** — вычисление должно быть быстрым (иначе теряется смысл хэш-таблицы).

```javascript
// Простейшая хэш-функция для строк
function simpleHash(str, tableSize) {
    let hash = 0;
    for (let i = 0; i < str.length; i++) {
        hash = (hash + str.charCodeAt(i)) % tableSize;
    }
    return hash;
}
```

## Криптографические хэш-функции

**Криптографические хэш-функции** (SHA-256, MD5, BLAKE) обладают дополнительными свойствами:

- **Однонаправленность** — по хэшу невозможно восстановить исходные данные.
- **Стойкость к коллизиям** — невозможно подобрать два разных сообщения с одинаковым хэшом.
- **Лавинный эффект** — изменение одного бита входных данных сильно меняет хэш.

Вопрос использования криптографических хэшей в хэш-таблицах — это **компромис между производительностью и безопасностью
**.

**Почему обычно предпочитают некриптографические хэши (MurmurHash, xxHash, CityHash, FNV-1a):**

- Они значительно быстрее (в десятки раз).
- Для внутренних нужд программы (ключи не контролируются злоумышленником) достаточно хорошей равномерности.

**Когда нужны криптографические или рандомизированные хэши:**

- Хэш-таблица **торчит наружу** — обрабатывает ключи, поступающие от пользователя (HTTP-параметры, JSON-ключи, имена
  полей в форме).
- Злоумышленник может намеренно подобрать множество ключей с одинаковым хэшем, вызывая коллизии. Это приводит к
  деградации производительности хэш-таблицы до O(n) и может стать причиной **DoS-атаки**.

**Решения:**

1. Использовать криптографическую хэш-функцию (защита от коллизий, но медленно).
2. Использовать **рандомизированную хэш-функцию** с солью (случайное значение при запуске). Даже если сама функция не
   криптографическая, коллизии, подобранные для одного запуска, не сработают на другом.
3. Комбинировать: быстрый хэш + соль + защита от чрезмерно длинных цепочек (переключение на дерево при превышении
   порога, как в Java 8+).

**Пример из реального мира:** PHP долгое время использовал простой хэш, и это приводило к реальным DoS-атакам. Начиная с
PHP 5.3.9, ввели защиту — максимальный лимит на количество ключей в одной корзине, а также рандомизированный хэш. В
Python 3.3+ хэш строк рандомизирован по умолчанию (флаг `PYTHONHASHSEED`).

**Вывод:**

- Внутренние хэш-таблицы → некриптографические быстрые хэши.
- Публичные хэш-таблицы (пользовательские ключи) → рандомизированный хэш (или криптографический) + дополнительная защита
  от коллизий.

## Коллизии хэшей

**Коллизия** — ситуация, когда два разных ключа дают одинаковый индекс в массиве.

Причины:

- Количество возможных ключей бесконечно (или очень велико), а размер таблицы ограничен.
- Хэш-функция не идеальна.

Коллизии неизбежны из-за принципа Дирихле (принципа ящиков): если N ящиков и N+1 предмет, хотя бы в одном ящике окажется
два предмета.

## Разрешение коллизий: метод цепочек

**Метод цепочек (chaining)** — самый распространённый способ. В каждой ячейке массива хранится не одно значение, а
связный список (или другой контейнер) всех пар ключ-значение, попавших в эту ячейку.

```javascript
// Схематичная реализация
class HashTable {
    constructor(size = 100) {
        this.buckets = new Array(size).fill(null).map(() => []);
    }

    _hash(key) {
        let hash = 0;
        for (let i = 0; i < String(key).length; i++) {
            hash = (hash + String(key).charCodeAt(i)) % this.buckets.length;
        }
        return hash;
    }

    set(key, value) {
        const index = this._hash(key);
        const bucket = this.buckets[index];
        const existing = bucket.find(item => item.key === key);
        if (existing) {
            existing.value = value;
        } else {
            bucket.push({key, value});
        }
    }

    get(key) {
        const index = this._hash(key);
        const bucket = this.buckets[index];
        const item = bucket.find(item => item.key === key);
        return item ? item.value : undefined;
    }
}
```

## Хэш для составных типов

Для составных ключей (массивы, объекты) нужно хэшировать все компоненты. Например, для точки `(x, y)` можно вычислить
`hash = x * 31 + y`.

В JavaScript поведение хэширования для объектов определяется движком, но при использовании `Map` с объектами в качестве
ключей используется сравнение по ссылке, а не по содержимому.

## Потребление памяти и перехэширование

Хэш-таблица потребляет память:

- Массив бакетов (обычно занимает O(размер) памяти).
- Списки внутри бакетов (O(количество элементов)).

**Коэффициент заполнения (load factor)** = `количество элементов / размер массива`.

При достижении определённого порога (обычно 0.7–0.75 для цепочек, 0.5–0.6 для открытой адресации) производительность
падает. Решение — **перехэширование (rehashing)**:

1. Создать новый массив в 2 раза больше.
2. Перебрать все существующие пары ключ-значение.
3. Для каждой пары вычислить новый индекс в новом массиве.
4. Вставить.

Операция дорогая (O(n)), но происходит редко, амортизированная стоимость остаётся O(1) на операцию.

## Алгоритмическая сложность хэш-таблицы

| Операция | Средний случай | Худший случай                   |
|----------|----------------|---------------------------------|
| Поиск    | O(1)           | O(n) (все ключи в одной ячейке) |
| Вставка  | O(1)           | O(n)                            |
| Удаление | O(1)           | O(n)                            |

Худший случай возникает при плохой хэш-функции или атаке на подбор коллизий.

## Пример: атака DoS через подбор коллизий

Злоумышленник может подобрать множество ключей, дающих одинаковый хэш. Если сервер вставляет все эти ключи в
хэш-таблицу, они окажутся в одной ячейке, и сложность операций вырастет до O(n). При достаточно большом n сервер может
перестать отвечать.

**Защита:**

- Использовать криптостойкие хэши для публичных данных.
- Добавлять случайную **соль (salt)** при каждом запуске сервера.
- Использовать структуры с гарантированной сложностью (деревья) для критичных мест.

## Ключи должны быть сравнимыми для обработки коллизий

При разрешении коллизий (особенно в методе цепочек) недостаточно просто найти ячейку по индексу — нужно перебрать все
элементы в бакете и сравнить ключи на равенство. Поэтому хэш-таблица требует, чтобы ключи поддерживали операцию
сравнения на равенство.

## Соль (salt) для защиты от радужных таблиц

**Соль** — случайная строка, добавляемая к ключу перед хэшированием. В контексте хэш-таблиц соль предотвращает
предвычисленные атаки (например, подбор ключей, дающих коллизию).

При каждом запуске программы генерируется случайная соль, которая используется во всех хэш-вычислениях. Тогда коллизии,
подобранные для стандартной хэш-функции, не сработают.

## Другие способы разрешения коллизий

### Открытая адресация (open addressing)

При коллизии все элементы хранятся прямо в массиве, без внешних списков. Ищется другая свободная ячейка по определённому
правилу.

### Линейное пробирование (linear probing)

При коллизии проверяем следующую ячейку: `index = (index + 1) % size`. Повторяем, пока не найдём пустую.

**Проблема:** образуются кластеры (группы занятых ячеек), что замедляет поиск.

### Квадратичное пробирование (quadratic probing)

Шаг увеличивается квадратично: `index = (index + k²) % size`. Это уменьшает образование кластеров.

### Двойное хэширование (double hashing)

Используется вторая хэш-функция: `index = (index + step) % size`, где `step = secondHash(key)`. Это один из лучших
методов открытой адресации.

## Проблемы открытой адресации

- Требует тщательного удаления элементов (нельзя просто очистить ячейку — нужно маркировать как «удалено» или
  перестраивать кластер).
- Коэффициент заполнения должен быть низким (обычно < 0.5–0.6), иначе производительность резко падает.
- При удалениях образуются «дырки», ухудшающие поиск.

**Вывод:** метод цепочек проще и надёжнее для большинства задач.

|------------|----------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|

Использует рандомизированный хэш для защиты от DoS-атак (PYTHONHASHSEED).                                                                                                                                                                                                                                                                                                                                        |

Стандарт не специфицирует детали реализации, но все основные компиляторы (libstdc++, libc++) используют именно цепочки. Использует extendible hashing с иерархической структурой (directory + таблицы) для инкрементального роста. Более ранние версии (до 1.24) использовали классический метод цепочек с бакетами (по 8 пар ключ-значение на бакет) и оверфлоу-ссылками.                                                                                                                       |
| Rust       | `HashMap`            | **Swiss Table** через крейт `hashbrown`. Начиная с Rust 1.36 стандартная библиотека использует Swiss Table. Реализация портирована из Google SwissTable (abseil). Хэш-функция для строк — DJBX33A с рандомизацией для защиты от атак. Load factor ~0.5 для rehashing. |
| JavaScript | `Object`, `Map`      | См. следующий раздел.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |

**О Swiss Table:** Изначально разработана Google для C++ (abseil / Swiss Tables). Ключевые особенности:

- **Группы по 8 или 16 слотов** — контрольные байты хранят нижние 7 бит хэша (H2) для быстрой фильтрации.
- **SIMD-инструкции** — параллельное сравнение всех байт группы за одну операцию на x86_64 (до 16 слотов) и ARM.
- **Открытая адресация** с квадратичным пробированием.
- **Высокий load factor** — до 7/8 (87.5%) до начала rehashing.
- **Компактность** — 1 байт служебной информации на запись.

**В Go** реализация Swiss Table имеет особенности, связанные с инкрементальным ростом. Вместо одной таблицы используется
иерархия: directory указывает на несколько независимых таблиц, что позволяет расти без полного перехэширования всей
карты.

**В Rust** `std::collections::HashMap` — это обёртка над крейтом `hashbrown`, который является портом Swiss Table.
`hashbrown` также доступен как отдельный крейт для `no_std`-сред.

**О SwissTable (Rust):** Google SwissTable — это современная реализация хэш-таблицы с открытой адресацией, которая:

- Использует **SIMD-инструкции** для сканирования нескольких хэш-ячеек за одну операцию (до 16 ячеек одновременно на
  x86_64) [citation:1].
- Хранит **контрольные байты (control bytes)** отдельно от самих ключей/значений, что улучшает локальность памяти.
- Обеспечивает высокий коэффициент заполнения (до 7/8) при сохранении производительности.
- Потребляет всего **1 байт служебной информации на запись** (против 8 байт в старой реализации) [citation:1].

Rust не использует `hashbrown` напрямую, стандартный `std::collections::HashMap` — это обёртка над `hashbrown` с
собственным хэшером. При этом `hashbrown` доступен как отдельный крейт для `no_std`-сред.

## Объекты и ассоциативный доступ в JavaScript

Согласно спецификации ECMAScript, объекты в JavaScript — это **ассоциативные массивы** (сопоставление ключей со
значениями). При этом **любой объект** является таким ассоциативным массивом, просто способы доступа могут различаться:

- Обычные объекты `{}` — используют скобочную нотацию `obj[key]` или точечную `obj.key` (для ключей-идентификаторов).
- `Map` — тоже объект (экземпляр класса `Map`), предоставляющий ассоциативное отображение через методы `get(key)` и
  `set(key, value)`.
- `Set` — объект для хранения уникальных значений (ассоциативный массив, где значение равно ключу).
- Функции, массивы, встроенные объекты — также являются ассоциативными массивами и могут иметь произвольные свойства.

```javascript
// Обычный объект — скобочная нотация
const obj = {name: "Иван"};
console.log(obj["name"]); // "Иван"

// Map — тоже объект, но с интерфейсом через методы
const map = new Map();
map.set("name", "Иван");
console.log(map.get("name")); // "Иван"

// У Map тоже можно добавлять свойства как у обычного объекта
// (но так делать не рекомендуется — это внутреннее хранилище)
map.custom = "значение";
console.log(map.custom); // "значение"

// Set — объект для уникальных значений
const set = new Set();
set.add("Иван");
console.log(set.has("Иван")); // true
```

### Что говорит спецификация

Объект в ECMAScript — это коллекция **свойств (properties)**. Каждое свойство имеет:

- Ключ (только `string` или `Symbol`)
- Значение
- Набор атрибутов (writable, enumerable, configurable)

`Map` и `Set` не хранят свои пары ключ-значение как свойства объекта (они не перечисляются в `Object.keys()` и не видны
через скобочную нотацию). Внутри у них есть **слоты** (спецификационные внутренние поля, например `[[MapData]]`),
которые хранят данные. Но сами экземпляры `Map` и `Set` остаются объектами, и к ним можно добавлять обычные свойства.

Спецификация определяет **абстрактные операции** для работы с объектами: `Get`, `Set`, `HasProperty`, `Delete` и т.д.
Эти операции работают с любыми объектами, включая `Map` и `Set`, но для доступа к данным `Map` используются методы, а не
стандартные `Get`/`Set` свойств.

### Как реализовано на практике

Хотя спецификация говорит только про ассоциативное поведение, все современные движки (V8, SpiderMonkey, JavaScriptCore)
используют **хэш-таблицы** как основу для хранения свойств объектов. Однако с мощными оптимизациями:

1. **Скрытые классы (Hidden Classes)** — для объектов с фиксированной структурой.
2. **Словарный режим (Dictionary Mode)** — для динамических объектов с частым добавлением/удалением свойств.
3. **Раздельное хранение** — числовые индексы массивов могут храниться в отдельном векторе (backing store).
4. **Инлайн-свойства (in-object properties)** — часть свойств хранится прямо в самом объекте, без обращения к
   хэш-таблице.

Для `Map` и `Set` движки используют специализированные структуры данных (обычно тоже хэш-таблицы), но отдельные от
механизма свойств объектов. Это позволяет оптимизировать их под свои задачи (например, быстрый перебор в порядке
вставки).

### Разные структуры для разных целей

В JavaScript есть несколько встроенных ассоциативных структур (все они — объекты):

| Структура | Интерфейс       | Типы ключей             | Порядок                                      | Прототип         | Когда использовать            |
|-----------|-----------------|-------------------------|----------------------------------------------|------------------|-------------------------------|
| `Object`  | скобки / точка  | строки, символы         | сложный (целые индексы + порядок добавления) | есть             | Простые объекты, JSON         |
| `Map`     | `get()`/`set()` | любые                   | порядок вставки                              | есть (но пустой) | Динамические словари          |
| `WeakMap` | `get()`/`set()` | только объекты (слабые) | не перечисляем                               | есть             | Кэши, приватные данные        |
| `Set`     | `add()`/`has()` | любые                   | порядок вставки                              | есть             | Уникальные значения           |
| `WeakSet` | `add()`/`has()` | только объекты (слабые) | не перечисляем                               | есть             | Множества объектов без утечек |

**Вывод:** все перечисленные структуры — это объекты в смысле спецификации. Разница — в способе доступа к ассоциативным
данным (скобочная нотация у обычных объектов против методов у `Map`/`Set`) и во внутреннем устройстве хранения.

## Объекты и ассоциативный доступ в JavaScript

Согласно спецификации ECMAScript, объекты в JavaScript — это **ассоциативные массивы** (сопоставление ключей со
значениями). Ключи могут быть только строками или символами. При этом спецификация **не предписывает** конкретную
реализацию — она описывает только требуемое поведение.

Важно понимать, что **любой объект** в JS является таким ассоциативным массивом:

- Обычные объекты `{}`
- Функции (у них тоже могут быть свойства)
- Массивы (числовые индексы + дополнительные свойства)
- Встроенные объекты (`Date`, `RegExp` и т.д.)
- `Map` и `Set` (но у них другая семантика ключей)

```javascript
// Любой объект — ассоциативный массив
const obj = {name: "Иван"};

const func = function () {};
func.customProperty = "значение";

const arr = [1, 2, 3];
arr.custom = "свойство массива";

console.log(obj["name"]);      // "Иван"
console.log(func["customProperty"]); // "значение"
console.log(arr["custom"]);    // "свойство массива"
```

### Что говорит спецификация

Объект в ECMAScript — это коллекция **свойств (properties)**. Каждое свойство имеет:

- Ключ (только `string` или `Symbol`)
- Значение
- Набор атрибутов (writable, enumerable, configurable)

Спецификация определяет **абстрактные операции** для работы с объектами: `Get`, `Set`, `HasProperty`, `Delete` и т.д.
Она не требует, чтобы эти операции были реализованы через хэш-таблицу — это деталь реализации движка.

### Как реализовано на практике

Хотя спецификация говорит только про ассоциативное поведение, все современные движки (V8, SpiderMonkey, JavaScriptCore)
используют **хэш-таблицы** как основу для хранения свойств объектов. Однако с мощными оптимизациями:

1. **Скрытые классы (Hidden Classes)** — для объектов с фиксированной структурой.
2. **Словарный режим (Dictionary Mode)** — для динамических объектов с частым добавлением/удалением свойств.
3. **Раздельное хранение** — числовые индексы массивов могут храниться в отдельном векторе (backing store).
4. **Инлайн-свойства (in-object properties)** — часть свойств хранится прямо в самом объекте, без обращения к
   хэш-таблице.

```javascript
// Обычный объект — на практике может быть реализован через скрытый класс
const obj = {
    name: "Иван",
    age: 30,
    city: "Москва"
};

// Добавление свойства может переключить объект в другой режим
obj.job = "инженер";
```

### Разные структуры для разных целей

В JavaScript есть несколько структур с ассоциативным доступом, каждая со своими характеристиками:

| Структура | Типы ключей             | Порядок                                      | Прототип | Когда использовать            |
|-----------|-------------------------|----------------------------------------------|----------|-------------------------------|
| `Object`  | строки, символы         | сложный (целые индексы + порядок добавления) | есть     | Простые объекты, JSON         |
| `Map`     | любые                   | порядок вставки                              | нет      | Динамические словари          |
| `WeakMap` | только объекты (слабые) | не перечисляем                               | нет      | Кэши, приватные данные        |
| `Set`     | любые (только ключи)    | порядок вставки                              | нет      | Уникальные значения           |
| `WeakSet` | только объекты (слабые) | не перечисляем                               | нет      | Множества объектов без утечек |

**Вывод:** хотя спецификация говорит об объектах как об ассоциативных массивах, а движки реализуют их через хэш-таблицы
с оптимизациями, важно понимать, что не всякое ассоциативное поведение в JS — это хэш-таблица. `Map`, `WeakMap`, скрытые
классы и инлайн-свойства — всё это примеры того, как реализация может отличаться от простой хэш-таблицы.

## Объекты как ассоциативные массивы в JavaScript

В JavaScript объекты — это, по сути, ассоциативные массивы, отображающие ключи (строки или символы) на значения. У
каждого объекта есть внутреннее хранилище свойств, которое движок может реализовывать различными способами (хэш-таблица,
скрытые классы, инлайн-свойства и т.д.).

```javascript
const obj = {
    name: "Иван",
    age: 30,
    "city": "Москва"
};

obj["job"] = "инженер";
console.log(obj.name); // "Иван"
```

### Примитивы и объекты-обёртки

Важно понимать различие между примитивами и объектами:

**Примитивные типы:** `string`, `number`, `boolean`, `null`, `undefined`, `symbol`, `bigint`.

Главные характеристики примитивов:

- **Неизменяемы (immutable)** — значение нельзя изменить, можно только создать новое.
- **Сравниваются по значению** (два примитива с одинаковым содержимым считаются равными, за исключением символов).
- Не имеют собственных свойств и методов (но при доступе к ним создаётся временная обёртка).

**Хранение примитивов:**

Вопреки распространённому упрощению, не все примитивы хранятся «непосредственно в переменной»:

- **Малые целые числа (SMI — Small Integers)** — хранятся прямо в ссылке (тегированный указатель) на стеке.
- **Дробные числа и большие целые** — хранятся в куче, переменная содержит ссылку на них.
- **Короткие строки** — могут храниться в стеке или внутри самой ссылки в зависимости от движка и длины.
- **Длинные строки** — всегда хранятся в куче, переменная содержит ссылку.

Однако независимо от способа хранения, **примитив остаётся примитивом** — он неизменяем и сравнивается по значению.

```javascript
// Все примитивы неизменяемы
let a = "hello";
let b = a;
b = b.toUpperCase(); // создаётся новая строка, старая не изменилась
console.log(a); // "hello" — оригинал не изменился

let c = 1.5;
let d = c;
d = d + 0.5; // создаётся новое число
console.log(c); // 1.5

// Сравнение по значению
console.log("hello" === "hello"); // true
console.log(10 === 10.0); // true
```

### Доступ к свойствам примитивов

При доступе к свойству примитива (например, `"hello".length`) JavaScript **временно создаёт объект-обёртку**, вызывает
метод, а затем уничтожает обёртку. Это позволяет примитивам «вести себя» как объектам, но без постоянного выделения
памяти.

```javascript
const str = "hello";
console.log(str.length);     // 5 — создаётся временная обёртка String
console.log(str.toUpperCase()); // "HELLO"

str.custom = "value";        // создаётся обёртка, свойство добавляется к ней
console.log(str.custom);     // undefined — обёртка уже уничтожена

const num = 42.5;
console.log(num.toFixed(2)); // "42.50" — обёртка Number

const bool = true;
console.log(bool.toString()); // "true" — обёртка Boolean
```

### Объекты-обёртки

`new String()`, `new Number()`, `new Boolean()` создают полноценные объекты, которые хранят значение и могут иметь
свойства. Однако их использование на практике **не рекомендуется** — они медленнее и могут приводить к неожиданному
поведению.

```javascript
const primitive = "hello";
const wrapped = new String("hello");

console.log(typeof primitive); // "string"
console.log(typeof wrapped);   // "object"
console.log(primitive === wrapped); // false — разные типы

wrapped.custom = "value";
console.log(wrapped.custom);   // "value" — обёртка сохраняет свойство
```

**null и undefined** не имеют объектов-обёрток — попытка доступа к их свойствам вызывает ошибку.

```javascript
null.toString();  // TypeError: Cannot read properties of null
undefined.foo;    // TypeError: Cannot read properties of undefined
```

## Объекты и прототипы

Объекты в JS имеют **прототипное наследование**. При обращении к свойству:

1. Сначала ищется своё свойство.
2. Если не найдено — поиск идёт по цепочке прототипов.

Это позволяет экономить память (методы не копируются в каждый объект), но добавляет сложность при проверке наличия
ключа (`hasOwnProperty`).

```javascript
const parent = {inherited: "value"};
const child = Object.create(parent);
child.own = "own value";

console.log(child.inherited); // "value" — из прототипа
console.log(child.own); // "own value"
console.log(child.toString); // из прототипа Object
```

## Порядок обхода свойств объекта

В современном JS порядок перечисления собственных строковых свойств объекта (через `for...in`, `Object.keys()`,
`Reflect.ownKeys()`) следует правилу:

1. Целочисленные индексы (в порядке возрастания).
2. Строковые ключи (в порядке добавления).
3. Символьные ключи (в порядке добавления).

Это важно, потому что классическая хэш-таблица не гарантирует порядка, а JS-объекты — гарантируют.

## Дескрипторы свойств объекта

Каждое свойство объекта имеет **дескриптор**:

- `value` — значение.
- `writable` — можно ли изменять.
- `enumerable` — появляется ли при `for...in`.
- `configurable` — можно ли удалить или изменить дескриптор.

```javascript
Object.defineProperty(obj, "secret", {
    value: "123",
    writable: false,
    enumerable: false,
    configurable: false
});
```

## Реализация объектов в JS

Внутренняя реализация объектов в движках (V8, SpiderMonkey) гораздо сложнее простой хэш-таблицы. Они используют
несколько стратегий:

- **Скрытые классы (Hidden Classes)** — для объектов с фиксированной структурой.
- **Словари (Dictionary mode)** — для объектов, у которых динамически добавляются/удаляются свойства.

## Оптимизация через скрытые классы

**Скрытые классы** (также известны как «карты» или «формы») — ключевая оптимизация в V8. Идея:

- Объекты с одинаковым набором свойств (и в одинаковом порядке добавления) разделяют один скрытый класс.
- Скрытый класс хранит смещения свойств в памяти (как в структуре C).
- Доступ к свойству превращается в доступ по фиксированному смещению — очень быстро.

```javascript
// Объекты point1 и point2 разделяют скрытый класс
const point1 = {x: 10, y: 20};
const point2 = {x: 30, y: 40};

// Добавление нового свойства создаёт новый скрытый класс
point1.z = 50; // теперь у point1 и point2 разные скрытые классы
```

**Правила для эффективного использования скрытых классов:**

- **Инициализируйте все свойства в конструкторе** — это создаёт стабильную структуру. Если вы заранее знаете все
  свойства объекта, объявите их в конструкторе (даже со значениями `null` или `undefined`). Это позволяет движку создать
  скрытый класс с фиксированным набором свойств.

- **Добавляйте свойства в одном и том же порядке** — если вы создаёте несколько объектов с одинаковым набором свойств,
  но в разном порядке, они получат разные скрытые классы, и оптимизация не сработает.

```javascript
// Хорошо: одинаковый порядок
const p1 = {x: 10, y: 20};
const p2 = {x: 30, y: 40};
// p1 и p2 разделяют скрытый класс

// Плохо: разный порядок
const p3 = {y: 10, x: 20};
const p4 = {x: 30, y: 40};
// p3 и p4 имеют разные скрытые классы
```

- **Добавление новых свойств после создания объекта возможно** — это не запрещено и не «ломает» оптимизацию мгновенно.
  Однако каждое добавление нового свойства создаёт **новый скрытый класс** в цепочке. При частых добавлениях цепочка
  становится длинной, что снижает эффективность.

```javascript
const point = {x: 10, y: 20};  // скрытый класс C0
point.z = 30;                     // новый скрытый класс C1 (производный от C0)
point.color = "red";              // новый скрытый класс C2 (производный от C1)
// Доступ к свойствам всё ещё быстрее словаря, но уже не так эффективен
```

- **Избегайте `delete`** — удаление свойства переводит объект в «словарный режим» (slow mode), где доступ к свойствам
  осуществляется через хэш-таблицу. После этого вернуться к скрытым классам невозможно.

```javascript
const obj = {a: 1, b: 2, c: 3};
delete obj.b;  // объект переходит в словарный режим, оптимизация теряется
```

- **Не смешивайте типы в одном свойстве** — хранение значений разных типов в одном и том же свойстве в разных объектах
  также может создавать разные скрытые классы.

```javascript
// Хорошо: тип свойства одинаков
const p1 = {x: 10};
const p2 = {x: 20};

// Плохо: типы разные
const p3 = {x: 10};
const p4 = {x: "20"}; // другой скрытый класс
```

**Как работает цепочка скрытых классов:** V8 создаёт скрытые классы постепенно. При добавлении первого свойства
создаётся C0 (пустой) и затем C1 (с первым свойством). При добавлении второго свойства — C2 (с двумя свойствами) и так
далее. Объекты с одинаковой историей добавления свойств разделяют скрытые классы.

## Когда нужна хэш-таблица

Хэш-таблица — идеальный выбор, когда:

- Нужно хранить пары ключ-значение с произвольными ключами.
- Ключи — строки, числа или объекты (с корректными equals/hash).
- Важен быстрый доступ (O(1) в среднем).
- Порядок хранения не важен.

Когда НЕ нужна:

- Ключи — последовательные целые числа (лучше обычный массив).
- Нужен отсортированный порядок ключей (лучше дерево).
- Очень мало элементов (константные накладные расходы могут быть выше, чем простая пара массивов).

## Массивы: внешний интерфейс (объект) + внутренний (вектор)

В JavaScript массив — это объект с числовыми индексами, но внутренняя реализация сложнее и зависит от движка.

**Векторное хранение (backing store):**

Независимо от наличия дырок, большинство современных движков (V8, SpiderMonkey) хранят элементы массива в **векторе** —
непрерывном буфере памяти, как у C++-подобных массивов. Разница лишь в том, что для «дырок» используется специальное
значение-маркер (`hole`).

```javascript
// Плотный массив — все слоты заполнены
const dense = [1, 2, 3, 4];
// Вектор: [1, 2, 3, 4]

// Разреженный массив с дыркой
const sparse = [1, , 3];
// Вектор: [1, hole, 3]  — память всё равно выделена под 3 слота
// hole — специальное значение, означающее "пусто"
```

**Когда массив может перестать быть вектором:**

- **Слишком большая дырявость** — если массив имеет очень большой размер, но лишь несколько заполненных элементов (
  например, `arr[1000000] = 42`), движок может переключиться на хэш-таблицу, чтобы не выделять огромный непрерывный
  буфер под несколько значений.
- **Смешанный доступ к массиву как к объекту** — добавление нечисловых свойств ( `arr["custom"] = "value"`) обычно не
  влияет на хранение числовых элементов, но движок может перейти в более медленный режим.
- **Типизированные массивы (`Uint8Array` и др.)** — всегда хранятся как непрерывный буфер фиксированного размера, без
  переключения режимов.

```javascript
// Экстремально разреженный массив — может переключиться на хэш-таблицу
const huge = [];
huge[1000000] = 42;  // выделять миллион слотов ради одного значения неэффективно
```

**Вывод:** в большинстве случаев массивы в JS хранятся как векторы. Дырки не приводят к автоматическому переключению на
хэш-таблицу — используется маркер `hole`. Переход на альтернативные структуры происходит только при очень сильной
разреженности или после определённых операций, нарушающих оптимизации движка.

## Map и Set

ES2015 ввёл `Map` и `Set` — специализированные структуры для ассоциативных коллекций, которые во многих сценариях
удобнее и предсказуемее обычных объектов.

### Проблемы объектов как ассоциативных массивов

Обычные объекты JavaScript имеют ряд особенностей, которые делают их неидеальными для использования в качестве чистых
словарей:

**1. Наследование от прототипа**

Объекты наследуют свойства из цепочки прототипов. Это значит, что даже пустой объект уже содержит встроенные свойства и
методы, такие как `toString`, `valueOf`, `constructor` и другие.

```javascript
const dict = {};
console.log(dict["toString"]); // function toString() { [native code] }
// Ключ "toString" существует, даже если вы его не добавляли!
```

Это приводит к неожиданным коллизиям ключей. Если пользователь введёт ключ `"toString"`, `"valueOf"`,
`"hasOwnProperty"`, он может случайно переопределить встроенные методы или получить неожиданное поведение.

**2. Символы как ключи**

Символы — уникальные и не перечисляемые ключи. Они не конфликтуют со строковыми ключами, но могут усложнить итерацию и
отладку.

```javascript
const sym = Symbol("id");
const obj = {
    [sym]: 42,
    normal: "value"
};

console.log(Object.keys(obj)); // ["normal"] — символы не попадают
console.log(Object.getOwnPropertySymbols(obj)); // [Symbol(id)]
// Нужно два разных метода, чтобы получить все ключи
```

**3. Непредсказуемый порядок итерации**

Хотя современные спецификации определяют порядок перечисления строковых свойств объекта, он сложный и неинтуитивный:

- Целочисленные индексы (в возрастающем порядке).
- Строковые ключи (в порядке добавления).
- Символьные ключи (в порядке добавления).

```javascript
const obj = {};
obj["2"] = "два";
obj["3"] = "три";
obj["1"] = "один";
obj["a"] = "а";
obj["b"] = "б";

console.log(Object.keys(obj)); // ["1", "2", "3", "a", "b"]
// Числовые индексы всегда идут первыми и сортируются!
```

### Почему Map проще и предсказуемее

`Map` решает все эти проблемы:

**1. Нет прототипа**

`Map` не наследует свойства от прототипа. Ключи живут в изолированном хранилище и не конфликтуют с методами самого `Map`.

```javascript
const map = new Map();
console.log(map.get("toString")); // undefined — нет конфликта
map.set("toString", "кастомное значение");
console.log(map.get("toString")); // "кастомное значение" — работает предсказуемо
```

**2. Равная работа со всеми типами ключей**

`Map` одинаково хорошо работает со строками, числами, символами и объектами, используя единый алгоритм сравнения `SameValueZero`.

```javascript
const map = new Map();
map.set(1, "число");
map.set("1", "строка");
map.set(Symbol("id"), "символ");
console.log(map.size); // 3 — все ключи различны
```

**3. Гарантированный порядок вставки**

`Map` сохраняет порядок, в котором ключи были добавлены, без каких-либо исключений. Никакой специальной обработки для числовых индексов.

```javascript
const map = new Map();
map.set("2", "два");
map.set("3", "три");
map.set("1", "один");
map.set("a", "а");
map.set("b", "б");

console.log([...map.keys()]); // ["2", "3", "1", "a", "b"] — порядок вставки
```

**4. Простая итерация**

`Map` предоставляет единые и предсказуемые методы для итерации: `keys()`, `values()`, `entries()` и поддержку
`for...of`.

### Другие преимущества Map

- **Производительность** — при частом добавлении и удалении свойств объекты могут переключаться в медленный «словарный
  режим». `Map` оптимизирован для динамических операций и сохраняет стабильную производительность.
- **Любые типы ключей** — объекты, функции, примитивы — всё может быть ключом в `Map`. Объекты требуют строковые или
  символьные ключи.
- **Размер коллекции** — у `Map` есть свойство `size`, которое легко получить. У объектов нужно использовать
  `Object.keys(obj).length`.
- **Безопасность** — отсутствие прототипных свойств исключает атаки через переопределение прототипа (`__proto__`).

### Когда использовать объекты

Несмотря на преимущества `Map`, обычные объекты остаются лучшим выбором в некоторых сценариях:

- **Работа с JSON** — объекты сериализуются и парсятся «из коробки». `Map` требует ручной конвертации.
- **Фиксированная схема** — когда набор свойств известен заранее и не меняется. Объекты могут использовать оптимизации
  скрытых классов.
- **Краткость и читаемость** — для простых статических структур точечная нотация (`obj.key`) удобнее вызовов методов (
  `map.get("key")`).
- **Передача в API** — многие встроенные функции и библиотеки ожидают обычные объекты.

| Критерий                                 | Object                           | Map                                      |

## WeakMap и WeakSet

**WeakMap** и **WeakSet** — варианты Map и Set с «слабыми» ссылками на ключи (WeakMap) или значения (WeakSet).

Особенности:

- Ключи — только объекты (не примитивы).
- Ссылки на ключи не предотвращают сборку мусора.
- Не перечисляемы (нет `keys()`, `values()`, `forEach`).
- Автоматическое удаление записей, если объект-ключ уничтожен.

```javascript
let obj = {data: "секрет"};
const wm = new WeakMap();
wm.set(obj, "важное значение");

obj = null; // объект может быть собран GC
// запись в WeakMap автоматически исчезнет
```

**Применение:** кэширование, хранение приватных данных, предотвращение утечек памяти.

## Сложности в JS из-за дизайна языка

Особенности JavaScript, которые нужно помнить:

1. **Объекты — это одновременно и ассоциативные массивы, и основа ОО-парадигмы.** С одной стороны, объект можно использовать как словарь (динамически добавлять/удалять свойства). С другой — это единица ООП с прототипным наследованием, методами, конструкторами и скрытыми классами. Совмещение этих ролей создаёт компромиссы.

2. **Оптимизация через скрытые классы** — это не просто «фича для производительности», а фундаментальный механизм, обеспечивающий эффективную работу объектной модели JavaScript. Без скрытых классов каждый доступ к свойству был бы полноценным поиском в хэш-таблице, что неприемлемо медленно. Однако скрытые классы требуют дисциплины от программиста для поддержания эффективности.

3. `delete obj.prop` — дорогая операция, переводит объект в медленный словарный режим. Для удаления лучше использовать `Map` или установить свойство в `undefined`.

4. `for...in` перечисляет перечисляемые свойства из прототипов — неожиданное поведение. Используйте `Object.keys()`, `Object.values()`, `Object.entries()`.

## Итоги лекции

- **Ассоциативный массив** — абстрактная структура для хранения пар ключ-значение.
- **Хэш-таблица** — основная реализация ассоциативного массива. Преобразует ключ в индекс через хэш-функцию.
- **Коллизии** неизбежны. Основные способы разрешения:
  - **Метод цепочек** — связный список в каждой ячейке.
  - **Открытая адресация** — поиск другой свободной ячейки.
- **Перехэширование** — увеличение размера таблицы при достижении порога заполнения.
- Средняя сложность операций — O(1), худшая — O(n) (при плохой хэш-функции или атаке).
- **В JavaScript**:
  - `Object` — это хэш-таблица со строковыми/символьными ключами, прототипами и сложными правилами перечисления.
  - Движки (V8) оптимизируют объекты через **скрытые классы** — доступ по смещению, как в структурах C.
  - Для чистых словарей лучше использовать `Map` (любые ключи, порядок вставки, отсутствие прототипов).
  - `WeakMap` / `WeakSet` — для кэширования с автоматической сборкой мусора.
- **Правила производительности**:
  - Инициализируйте все свойства в конструкторе.
  - Не удаляйте свойства (избегайте `delete`).
  - Используйте `Map` для динамических коллекций.
